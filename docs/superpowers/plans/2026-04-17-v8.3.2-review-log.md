## New-table reader audit (mandatory per rule)

**Important correction to plan text:** the plan references a migration
`005_streaming_audio.sql` that "adds `audio_path` column to
`streaming_segments`". There is no such migration file on disk. The
`streaming_segments` table was defined by `data-migrations/003_streaming_segments.sh` with an `audio_path TEXT` column already present; that column was added to `003` during the v8.3.1 audit (commit `9faa9f5`, "phase-7 iterative re-audit fixes — F1..F18"), before this feature branch forked. The column is therefore established DDL on any v8.3.1-or-later database, and v8.3.2 is the first release that actually uses it.

That is a CHANGELOG bookkeeping bug (see Findings), not a release
blocker. What the rule cares about — "a running process reads AND writes
this column" — is the real test, and both ends of the pair exist:

- **Writer (worker)**: `scripts/stream-translate-worker.py:371` POSTs
  `{"vtt_content": …, "audio_path": audio_rel, …}` to the
  `/api/translate/streaming/segment/complete` endpoint after edge-tts
  synthesizes the opus file.
- **Writer (API)**: `library/backend/api_modular/streaming_translate.py:918-928`
  implements `segment_complete_impl` → `UPDATE streaming_segments SET
  vtt_content = ?, audio_path = ?, completed_at = CURRENT_TIMESTAMP
  WHERE …` — this is the DB write triggered by the worker POST above.
- **Writer (API, chapter-level)**: same file at line 1007-1012 inserts
  into `chapter_translations_audio` when all segments for a chapter
  have `audio_path` populated.
- **Reader (API consolidator)**: same file at line 1045 — `SELECT
  segment_index, audio_path FROM streaming_segments WHERE …` drives
  the `ffmpeg -f concat` consolidation that produces the
  chapter-level opus for offline cache.
- **Reader (API route)**: same file at line 1247 — the
  `/streaming-audio/<int:audiobook_id>/<int:chapter_index>/<int:segment_index>/<locale>`
  route serves the per-segment opus bytes via `send_file` (Range-aware)
  for the browser MSE chain. The route joins on `audio_path` as the
  filesystem key.
- **Reader (client)**: `library/web-v2/js/streaming-translate.js:382 +
  443` constructs the `/streaming-audio/…` URL and feeds the response
  bytes into `MseAudioChain.enqueueSegment()` →
  `SourceBuffer.appendBuffer`.

**Result:** the write/read pair is closed end-to-end. No v8.3.1-style
orphan table.
