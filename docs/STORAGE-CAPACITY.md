# Storage Capacity — NVMe Root Filesystem

> **Observed 2026-05-09 during /test Phase 4b audit.** NVMe (`/dev/nvme0n1p2`, 927 GB)
> was at **100% used (2.9 GB free)**. This document captures the top consumers
> and proposes retention policies.

| Path | Size | Notes |

| ├─ `overlay2/` | 93 GB | Image layer union mounts |

## Key Finding: Docker Build Cache (Reclaimable)

`docker system df` shows **18.09 GB of build cache** marked 100% reclaimable.
Safe to prune without affecting running containers:

```bash
sudo docker builder prune --keep-storage 1gb --force
```

Also: **47.12 GB of Docker images** (76%) are reclaimable — these are dangling or
untagged images from previous builds. To prune safely (keeps images used by running
containers):

```bash
sudo docker image prune -f
```

**Do not prune volumes** — `the_bosco_club_immich-pgdata` and
`the_bosco_club_mariadb-data` contain live service data.

## Streaming-Audio Retention Policy (Proposed)

`/var/lib/audiobooks/streaming-audio/` stores translated WebM audio segments
generated by the stream-translate worker. These accumulate permanently with no
current expiry.

**Observed pattern**:
- 1,877 unique book directories, 27,914 `.webm` files, 5.2 GB total
- Oldest files: 2026-04-23 (16 days old at observation time)
- Newest files: 2026-05-04

**Proposed retention rule** (to be implemented in a future release):

Delete segments for a book when ALL of the following are true:
1. All chapters of that book have a completed `chapter_translations_audio` row
   (i.e., the full `chapter.webm` concatenation exists)
2. The book has not been accessed (no streaming playback) in the past **90 days**
3. The per-chapter `chapter.webm` files themselves are retained (safe to
   regenerate from segments, but regeneration costs GPU time)

Alternatively: keep only the per-chapter `chapter.webm` concatenations and delete
the constituent `seg*.webm` files once a chapter is complete. The player streams
from segments during live translation but falls back to `chapter.webm` on replay.

**Implementation location**: `library/backend/api_modular/maintenance_tasks/` —
a new `StreamingAudioCleanupTask` that runs as part of the maintenance scheduler.

**Manual interim cleanup** (operator discretion — deletes no chapter.webm):
```bash
# Preview what would be deleted: seg*.webm older than 90 days
sudo find /var/lib/audiobooks/streaming-audio/ -name "seg*.webm" \
  -mtime +90 -type f | wc -l

# Actual deletion (confirm preview count first)
sudo find /var/lib/audiobooks/streaming-audio/ -name "seg*.webm" \
  -mtime +90 -type f -delete
```

## Auth DB Upgrade Backups

`/var/lib/audiobooks/auth.db.pre-upgrade-*` files (two copies, 2 MB each) are
created by `upgrade.sh` before each schema migration. These are safe to delete
once the upgrade is confirmed stable. After keeping the most recent backup:

```bash
# Keep most recent, delete older ones
sudo ls -t /var/lib/audiobooks/auth.db.pre-upgrade-* | tail -n +2 | sudo xargs rm -f
```

## NVMe Capacity Planning

The 927 GB NVMe holds the root filesystem (`/`), application state
(`/var/lib/audiobooks/`), Docker data (`/var/lib/docker/`), and user home
(`/home/`). At current growth rates:

- Docker images: stable if old images are pruned after each release
- `streaming-audio/`: grows at ~320 MB/book (average), linear with translation activity
- Auth DB backups: 2 MB × number of upgrades — cap at 3 retained copies

**Recommended action before next release**: `docker builder prune` + `docker image prune`.
This recovers ~65 GB without touching running services.
